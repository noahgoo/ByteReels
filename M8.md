# M8 — Feed Controls & Playback Enhancements

**Status:** Planning  
**Date:** 2026-04-07

---

## Overview

Four self-contained features that improve how users control and discover content in the feed. Each can be built and shipped independently.

---

## Features

### 1. Speed Filter

**Goal:** Let users filter the feed by video duration so they can match content to how much time they have.

**Behaviour:**
- Add a second filter control (alongside the existing topic dropdown) that lets the user pick a duration bucket:
  - **Any** (default — current behaviour)
  - **Quick** — under 2 minutes
  - **Short** — 2–5 minutes
  - **Medium** — 5–10 minutes
- Filtering is client-side only — no additional API calls. All videos are already fetched with `durationSeconds`.
- The active duration filter persists to `localStorage` (`bytereels_speed_filter`) across sessions.
- Combining speed filter + topic filter narrows the feed (AND logic).

**UI:** A pill button or dropdown in the filter bar row, consistent with the existing topic filter style.

**State:** Add `speedFilter` to `feedStore` alongside `activeFilter`. Computed `displayedVideos` in `SwipeFeed` applies both filters before the stable sort.

---

### 2. Shuffle Mode

**Goal:** Randomize the feed order so repeated sessions feel fresh and users don't always see the same videos first.

**Behaviour:**
- A toggle in the filter bar row (or header). When on, the feed is shuffled on load and on every manual refresh.
- Shuffle is applied after filtering (topic + speed) but before the watched-sink sort — watched videos still float to the bottom.
- The shuffled order is stable for the session (doesn't re-shuffle as the user swipes).
- Shuffle state does **not** persist across sessions — it resets to off on next open.

**UI:** A small shuffle icon button in the header or filter row. Active state shown with an accent colour (orange to match skip buttons).

**State:** Add `shuffleEnabled` boolean to `feedStore`. The sort step in `SwipeFeed` uses a seeded shuffle when enabled (seed set once on toggle, stored in a ref so swipes don't re-shuffle).

---

### 3. Playback Speed Control

**Goal:** Let users watch at faster or slower speeds — especially useful for tutorials where 1.5× or 2× saves time.

**Behaviour:**
- A speed button on the active video card (e.g. bottom-left, near the skip controls).
- Tapping cycles through speeds: **1×** → **1.25×** → **1.5×** → **2×** → **0.75×** → **1×**
- Uses YouTube IFrame Player API `setPlaybackRate(rate)`.
- The button label shows the current rate (e.g. `1.5×`).
- Speed resets to 1× when the user swipes to the next video.
- Speed does **not** persist across sessions.

**Implementation:** `VideoCard` calls `embedRef.current?.setPlaybackRate(rate)` via a new method exposed through `useImperativeHandle` in `YouTubeEmbed`. The rate cycles in `VideoCard` local state.

**UI:** Small pill button, same style as skip controls, placed bottom-right alongside them. Label is the speed value only (`1×`, `1.5×`, etc.).

### 4. Share Button

**Goal:** Let users send a video link to someone else with one tap using the native iOS/Android share sheet.

**Behaviour:**
- A share icon button on each video card (or only the active card).
- Tapping calls `navigator.share({ title, url })` where `url` is `https://youtu.be/<videoId>`.
- Falls back to copying the URL to clipboard (`navigator.clipboard.writeText`) if the Web Share API is not available (desktop Chrome, etc.).
- A brief toast ("Link copied!") confirms clipboard fallback.

**Implementation:** `VideoCard` gets an `onShare` prop. `SwipeFeed` passes a handler that calls `navigator.share` or clipboard. No API calls, no backend.

**UI:** Share icon (standard iOS share box-with-arrow) in the video info section, or as an overlay button on the active card. Does not require `isActive`.
